8월

주요업무: 파악하기, 관찰하기

솔루션(프론트)부분과 엔진(백엔드)부분이 나누어져있고 매우 복합적으로 다들 신규개발, 유지보수등을 하고있었다.

특히 엔진 부분은 사이트에 로그인하고 크롤링하거나 API 없는 곳을 HTTP를 따와서 연동하거나 직접 브라우저를 띄워서 스크린스크래핑(화면제어)하는 일을 여러개 쇼핑몰마다 하고있었다.

나는 엔진을 개선하는 팀에 들어갔다. 로컬 또는 IDC서버에 C#으로 되어있는 사이트 연동 스크립트(엔진)이 엄청 많았다.

이걸 새로운 AWS 인프라에 node.js 기반으로 엔진을 만드는 업무를 맡았다.

엔진은 여러 엔진기능의 합집합이라서 엔진 개발을 개선한다는 개념이 모호했는데, 엔진기능 하나를 API라고 부르기로 하고 "쇼핑몰에 상품을 품절처리하는 API" 단위로 나누어서 다른 솔루션팀들이 쓰도록 정했다.

API이니 당연히 문서화, 버전관리, 모니터링 되겠지 생각함.

각 API에 필요한 추가정보(옵션데이터)는 기존에 솔루션(ENT, OMP, EMP, 2.0)에서 각자 다들 따로 만들고 있었는데. 옥션 한곳이 바뀌면 네 곳에 모두 필드를 고쳐야하는 문제가 있었다. 그래서 세트/템플릿 데이터를 별도 embeddable UI로 만들어 iframe, script 삽입으로 어떤 솔루션이건 추가정보를 달수있도록 했다. nodejs+mysql+angular1 아주기본

기존 레거시와의 호환성 유지 + 신규 솔루션과의 호환성 유지라는 과제가 있었다. 새로운걸 만드는 입장이었지만 기존의 것들을 습득하는데 많은 시간을 들여야했다. 새로운걸 만들더라도 원래걸 만드는 사람들의 인정을 받아야한다는 숙제가 있었다.

9월

주요업무: 작업환경에 초점을 두고 개발환경 개선하기

기존분들의 하소연을 듣다보니 생각이 들었다. 엔진개발 속도, 유지보수의 어려움, 교육의 어려움등등 이런건 C#, Node.js같은 프로그래밍 언어의 문제영역이라기보단 업무방식의 문제다. 더 나은 언어/프레임워크/플랫폼으로 이전함과 동시에 개발환경개선을 목표로 해야겠다 싶었다.

회사는 php4, apache(nginx), mysql, ftp, visual studio, C#, redmine등의 도구를 쓰고 있었다. (조금 오래되었다 뿐이지 나름 다들 한때 잘나가던 기술스택이다. 이밖에 레거시가 많은데 유지보수 업무가 바빠서 시간을 많이 못들인 느낌이었다)

다들 머리로는 공감하지만, 손으로는 공감못하는 개발도구/개발프로세스를 회사에 순차적으로 전파했다. git,github, pull reqeust, code review, CI/CD, staging. 이런건 백날 매뉴얼만들고 교육해도 소용이 없으니, CTO분 포함, 우리팀 자체적으로 (3명) 적극적 지지를 하며 행동으로 보여주고, 이걸 좋은 사례로 나머지에게 전파했다. (개인적으로 편하게쓰던 CoffeeScript와 SublimeText를 버리고 es6와 vscode로 옮겼다)

나는 입사한지 얼마안된 개발자이고 너무 급진적으로 진행하면 평화로운 조직을 파괴하는 역효과가 날까봐 조심했다. 실제로 이 부분에 대해선 아쉽다와 최선이었다가 공존한다.

10월

주요업무: 개선된 엔진개발환경 직접 써보기 - 개밥먹기

Github(브랜치 운영, 머지, rebase, PR), Wiki(README.md 또는 여기에 정리하기, FAQ 만들기), Github Issues(담당자 배정, 레이블 붙이기, 이슈 연결시키기, 커밋으로 닫기)등 협업환경에 힘썼다.

틈틈히 개발도구를 만들었다. debugs, polymatch등 회사에 도움되는 오픈소스를 몇개 공개했다. 작업요청/진행/실패정책/재시도/중간결과값등등에 대해 비동기작업 처리를 좀더 힘쓰고 싶었지만 생각보다 Job Queue(Message Queue)에 대해서 설득하지 못했다. 외부 연동 성격의 operation을 queue로 쌓아두고 처리하지않으면 내부요인 또는 외부요인으로 실패시 재시도가 어려워진다. 여기에 각 endpoint들의 이중화를 힘쓰고 싶었으나 설득하지 못했고, service discovery/register도 설득을 하지 못했다.

https://www.npmjs.com/package/debugs https://www.npmjs.com/package/polymatch

곧 오픈예정인 실제 신규솔루션에 5대 오픈마켓(옥션,지마켓,인터파크,11번가,네이버스토어팜-스마트스토어)와 종합몰등등 기본정보 API들을 제공하기 시작하면서 프로덕션 운영 부담이 늘었다. 5개 이상의 쇼핑몰 기본정보 API들을 자체적으로 2명이서 다 만들어보았는데 엄청 금방만들어서 다행이였다. 원래 계시던 분들에게도 새로운 엔진 환경을 설득할수있겠다 안도의 한숨을 쉬었다.

11월

주요업무: 새로운 엔진환경 전파하기, 부족한점 고치기

마무리 단계였다. Google Chrome의 devtools API를 활용한 puppeteer를 이용하여 nodejs로 화면제어를 구현했고, API마다 postman을 만들어 솔루션팀이 직접 테스트해보며 쓰게끔했고, API 모니터링은 runscope로 15분마다, 1시간마다등 테스트케이스를 계속 돌리며 초록불/빨간불/응답시간등을 확인했고, 로깅은 papertrail을 이용하여 stacktrace를 따라가 장애의 원점을 선대응하고자 노력했다. (5개 기본정보만 해도 100개 남짓 API들이라 시스템이 필요했다. 입사했을때 테스트코드를 거의 본적이 없는데, 이제 테스트코드도 많고 간단한 테스트 자동화까지하고 있어서 뿌듯했다)

다른 솔루션팀에서 필요한 신규엔진을 이걸로 하고싶다고해서 해당 개발자분 엔진개발 교육과 troubleshooting등을 진행했다. 크고작은 문제가 너무 많았다. 구현은 어떻게든 하면되는데 디자인적 결정이 어려웠다. (커스텀 관련 코드는 어디에 덧붙일지, 중복코드는 어디에다가 넣을지, 개념상 분리할지말지 등등, 테스트방법등등) 2명이서 매일 5명 정도를 응대를 했는데 조금 바빴다.

이때까지만 해도 엔진 core+api+쇼핑몰스크립트가 모두 한곳에 모여있어서 PR, 머지, 배포에 병목이 있었는데, 이걸 npm으로 나누기 시작했다. 자연스럽게 각 쇼핑몰마다 테스트코드도 나뉘어져 테스트 자체가 스펙의 역할을 하기 시작했다.

12월

주요업무: 개발/배포/정책/운영 걸림돌 치우고 bottleneck 나누기

엔진개발자를 위한 CLI를 만들었다. `npm install -g @playauto/pa`

`pa init`, `pa test`, `pa up` 등등 이게 없을때는 모든 엔진개발자들이 core+api server+쇼핑몰 모든코드 다 내려받아서 npm install하고 빌드한다음에 개발을 시작할수있었다. 당근 core+api server에러땜에 쇼핑몰 연동코드도 에러나고 아주 혼돈의 카오스.

추가적으로 각 엔진별 npm을 나누었다. 엔진마다 필요한 패키지(엑셀파싱, 문자열다루기, http request등등)이 많은데 아직 version이 1개이지만 언젠간 패키지하나 최신기능쓰려고 업데이트했다가 나머지들 우수수 장애나는 상황은 피하고 싶었다. 그리고 엔진하나 잘못만들었다고 전체시스템이 장애나는 상황도 피하고 싶었고. SPOF 분리.

NPM orchestration을 상상하며 만들고 missioncontrol이라고 불렀는데, 각 엔진(npm)마다 staging, production에 원하는 버전을 선택해올리고도록 QA/운영담당자용 만들었다. 개발자는 고치고 npm publish만 하면 특정버전으로 코드의 형상이 새로 생긴다. staging은 latest(항상최신버전) 선택, production은 QA가 끝났거나 각자 릴리즈 일정에 따라 알아서 상위버전을 선택한다. 그럼 각 서버들이 알아서 새엔진으로 교체한다. 이것도 지지난달 service discovery/register가 있어야 했는데 여전히 설득은 안되어서 그냥 redis pubsub을 이용했다. 개발과 배포의 시점을 나누는건 정말 중요하다!

시범용으로 2명이서 쿠팡, 위메프, 티켓몬스터 엔진을 만들었다. 실제 고객이 쓰기 시작한게 아니니 100% 완성이라고 볼수없다.

1월

주요업무: 안정화, 교육, 문서화

인프라, 배포의 문제로 장애가 계속되면서 정신없었다. 인력충원도 계속 실패해서 2명이서 20명정도 사람들을 케어하고 커뮤니케이션 했던것 같다. 계속 버그고치고 신규API 요청 받고 바쁘게 살다가.

갑자기 이제 2월부터 이 개발환경을 기존 개발팀(8명+2명)에게 전파하라고 명령 받았다. 의외로 윈도우즈 개발환경에 설정하는데에 어려움이 있었고 이제 큰 문제는 다 잡았다.

신규 솔루션이 곧 오픈한다길래 우리도 더이상 안고치고 stable version 만들기위해 노력했다.

쇼핑몰들을 연동하려면 다양한방식(API, request, 화면제어등등)을 RESTful HTTP API로 통일했고 + 제3자 공급이 가능하단걸 사업부도 알았기에, 우리 API를 돈주고 쓰려는 곳들이 찾아오는 미팅이 이어졌다.

개발환경에 대해 계속 문서화했다. 바빠서 더 까먹기전에 기록을 남겨두려고 했다. 이외에도 여러가지 이유로 점점 코딩하는 시간이 줄어들었다.

2월

기존 엔진 개발자분들이 거의 모두 node환경에서 엔진을 만들기 시작했다. 근본적인 문제를 해결하느랴 다소 시간을 많이 쓴거 아닌가? 생각도 들면서 어차피 겪을것 빨리 고생해버리자 느낌도 있었다. 6개월만에 이전하는것이니 너무 급한거 같으면서도 ... 아직 결과는 어떻게될지 모른다! 그다지 엔진 개선으로 잘했다 소리도 못들었고 경영진이나 사업부에서는 아직도 의심의 눈초리로 반신반의하는 상태이기 때문에 아직 갈길이 멀다! 우리팀 PLAYAPI TF - Matthew (https://github.com/480) - Jin (https://github.com/eces) - Kale (https://github.com/7vv)
